# Presentation Coach

練習録音（.wav / .mp3）と発表スライド（PDF またはテキスト）を入力に取り、理解が甘い領域と重点練習箇所をコーチングレポートとして出力する。

## 入力

| 入力 | 形式 | 必須 |
|------|------|------|
| 音声ファイル | `.wav` / `.mp3` | どちらか一方以上 |
| スライドファイル | PDF / テキスト | どちらか一方以上 |

PPTX・Google Slides など非対応形式の場合はユーザーに PDF またはテキストへの変換を依頼すること（このスキルではフォーマット変換を行わない）。

---

## Step 1: 音声分析（音声ファイルが提供された場合）

まず filler-cli が使えるか確認する。

```bash
command -v filler-cli >/dev/null 2>&1 \
  && echo "installed" \
  || echo "not found"
```

見つからない場合は以下をユーザーに案内して止まること:

```
filler-cli が見つかりません。
インストール: go install github.com/maro114510/filler-cli@latest
または Releases からバイナリをダウンロード: https://github.com/maro114510/filler-cli/releases

インストール後、AMIVOICE_API_KEY 環境変数を設定してから再実行してください。
```

filler-cli が使える場合は分析を実行する:

```bash
filler-cli analyze --format json <audio-file>
```

`AMIVOICE_API_KEY` が未設定の場合は対話プロンプトが起動する。API キーの取得先:
https://acp.amivoice.com/main/register/  （非 End-to-End エンジンのキーが必要）

### フィラー時間帯集計

JSON 出力の `durationSec` と `fillerEvents[].startMs` を使い、話全体を **前半 / 中盤 / 後半** の3等分に分けてフィラー数を集計する。

```
前半: startMs < durationSec * 1000 / 3
中盤: durationSec*1000/3 <= startMs < durationSec*1000*2/3
後半: startMs >= durationSec * 1000 * 2 / 3
```

各時間帯のフィラー密度（個/分）を算出し、全体平均の **1.5 倍以上** の時間帯を「要注意セクション」としてマークする。

---

## Step 2: スライド内容分析（スライドが提供された場合）

- **PDF**: `Read` ツールで直接読み込む
- **テキスト / Markdown**: そのまま読む
- **PPTX / Google Slides / その他**: ユーザーに変換を依頼する

読み込んだら以下を把握する:

1. **全体構成**: スライド総枚数と大きな流れ（導入・本論・まとめ）
2. **セクション分け**: 前半 / 中盤 / 後半の3グループに分けて各グループの主題を整理
3. **難易度マッピング**: 専門用語が多い・説明ステップが複雑・図解が必要なスライドを特定する

## 発表練習コーチングレポート

### 概要

| 項目 | 値 |
|------|----|
| 録音時間 | X 分 Y 秒 |
| フィラー総数 | N 個 |
| フィラー頻度 | X.X 個/分 |
| 最多フィラー | 「えーと」(N 回) など |

> フィラー頻度の目安: 1 個/分以下 → 優秀、2〜3 個/分 → 標準、4 個/分超 → 要改善

### フィラー分布（時間帯別）

| 時間帯 | フィラー数 | 密度 (個/分) | 評価 |
|--------|-----------|-------------|------|
| 前半（0〜X分） | N 個 | X.X | ◎ 良好 / △ 注意 / × 要改善 |
| 中盤（X〜Y分） | N 個 | X.X | ... |
| 後半（Y〜Z分） | N 個 | X.X | ... |

> ◎ 平均以下 / △ 平均〜1.5倍 / × 平均の1.5倍超

### スライド構成と難易度

前半 / 中盤 / 後半の各グループについて、以下の観点で難易度を分析する。

**分析観点（認知科学・教育工学の知見に基づく）**

- **内在的認知負荷**（Swellerの認知負荷理論）: 1枚のスライドに含まれる概念数・前提知識の深さ・概念間の依存関係の複雑さを評価する。相互依存する要素が多いほど、話し手のワーキングメモリに高い負荷がかかる。
- **分割注意効果**（Mayer のマルチメディア学習原則）: 図解・コード・箇条書きが画面上に散在していて、口頭で「どれを指しているか」を伝えながら話す必要があるスライドは詰まりやすい。
- **概念密度**: 1枚あたりの専門用語数・固有名詞数が多いスライドは、逐一定義を挿入する負担が大きい。
- **説明の方向性（Bloomの分類）**: 用語の「記憶・理解」レベルで済むスライドより、「分析・評価・創造」（例: なぜこのアーキテクチャを選ぶか）が求められるスライドの方が口頭での言語化が難しい。

各グループの主なトピックと、上記観点で特に難易度が高いスライドを1〜2文で説明する。

### 理解が甘い可能性のある領域（優先練習リスト）

時間帯フィラー集中とスライド難易度の両方を根拠に、練習優先度の高い領域を **3〜5 項目** 挙げる。
単なる「フィラーが多い」ではなく、**なぜそのトピックで詰まりやすいのか** の仮説を添える。

1. **[トピック名]** — [詰まりやすい理由の仮説]
2. ...

### 強みとして継続してよい点

流暢に話せているセクションや十分に練習できている領域を **1〜2 点** 挙げる。

### 次の練習セッションで意識すること（アクションアイテム）

フィラーの種類・タイミングを踏まえた、具体的なアクションを **3〜5 点** 挙げる。

例の形式:
- 「〇〇の説明に入る前に、一度止まって次の言葉を考えてから話す練習をする」
- 「スライド X の専門用語リストを声に出して読む練習を 5 回繰り返す」

## 制約事項

- 検出対象フィラーは AmiVoice が `%えーと%` 形式で返すトークンのみ。「まあ」「その」「はい」など文脈的フィラーは含まない ([KC-003](../../README.md#kc-003--manual-vs-api-count-discrepancy))。
- スライドと音声の精密な時刻対応付けは行わない。時間帯（前半 / 中盤 / 後半）の粒度で推定する。
- AmiVoice API は標準（非 End-to-End）エンジン必須。End-to-End エンジンはフィラートークンを抑制する ([KC-001](../../README.md#kc-001--filler-token-availability))。
